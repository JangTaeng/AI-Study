class AlexNet(nn.Module)에서 클래스는 단순히 코드를 묶는 문법이 아니라, 모델이 어떤 레이어들로 구성되어 있는지 기록하고, 그 레이어들의 가중치를 함께 관리하며, 입력이 들어왔을 때 어떤 순서로 계산할지를 정의하는 핵심 구조입니다.

__init__에서는 Conv, Linear 같은 레이어들을 만들어 self에 저장해두고, forward에서는 이미 만들어둔 그 레이어들을 순서대로 호출해 실제 계산을 수행합니다.

또한 nn.Module을 상속받기 때문에, 이렇게 저장된 레이어들은 PyTorch에 의해 자동으로 등록되어 model.parameters(), model.to(device), model.train(), model.eval() 같은 기능이 자연스럽게 동작합니다.

그래서 모델 코드에서 클래스는 단순한 “설계도”를 넘어서, 레이어·파라미터·실행 흐름을 모두 묶어서 실제로 동작하게 만드는 중심 구조입니다.

[클래스]  

데이터와 동작을 묶어서 관리하기 편하게 만드는 구조.  

클레스는, 상태관리, 멀티 디바이스, 멀티세션의 중요한 역할을함.

---
모델 코드에서 클래스는 쉽게 말해서 이미지를 넣으면 어떤 순서로 계산해서 결과를 낼지 정해둔 설계도입니다.  
내부에는 Conv, Linear 같은 레이어들이 순서대로 들어 있고, 각 레이어는 학습되는 값(가중치)을 가지고 입력을 조금씩 변환합니다.  
그리고 forward 함수는 이 레이어들을 어떤 순서로 통과할지를 정의하여 전체 계산 흐름을 고정합니다  
즉 이 클래스 하나가 데이터를 어떻게 처리할지, 어떤 값들을 사용할지, 어떤 순서로 계산할지까지 모두 포함한 모델 자체입니다.
---
그래디언트 누적 (Gradient Accumulation): 작은 배치를 여러 번 계산하여 오차(Gradient)를 쌓아둔 뒤, 큰 배치를 한 번에 학습한 것과 같은 효과를 내는 기법 /기존 방식: 100장 공부하고 → 바로 오답노트 쓰고 → 다음 100장 공부.

누적 방식: 25장 공부하고 오차 저장 → 또 25장 공부하고 오차 저장(총 50장분) → ... → 100장분이 모이면 그때 오답노트 작성. / 딥러닝 모델은 배치 사이즈(한 번에 보는 데이터 양)가 어느 정도 커야 학습이 안정적으로 잘 됩니다. 하지만 모델이 너무 크거나 내 GPU 메모리가 작으면, 큰 배치를 한꺼번에 넣었을 때 컴퓨터가 "메모리 부족!"이라며 뻗어버립니다.
이때 "가진 메모리 안에서 쪼개서 넣되, 결과는 크게 넣은 것처럼 만들자"는 아이디어가 나온 것입니다. / <작동 원리 (순차적 단계)> ① 1~3단계: 오차 수집 (Accumulate)

32장을 모델에 넣고 계산합니다.

이때 optimizer.zero_grad()를 호출하지 않습니다. (중요!)

loss.backward()를 실행하면, 오차가 지워지지 않고 기존 가중치에 계속 플러스(+) 됩니다.

② 4단계: 목표치 도달 및 업데이트 (Step)

32장씩 4번을 했으니 이제 128장치 오차가 모였습니다.

이제 드디어 optimizer.step()을 눌러서 모델을 실제로 업데이트합니다. (오답노트 작성 완료!)

③ 마지막: 청소 (Zero Grad)

업데이트가 끝났으니, 다음 128장을 위해 optimizer.zero_grad()를 호출해 오차 저장소를 깨끗이 비웁니다.
---
# 1) ✅ 일반 학습 (zero_grad 필요: 매 batch 독립 업데이트)
for x, y in loader:
    optimizer.zero_grad()      # 이전 grad 초기화
    loss = criterion(model(x), y)
    loss.backward()            # grad = 현재 batch 기준
    optimizer.step()           # 업데이트

# 2) ❌ 실수 (zero_grad 없음 → grad 누적됨)
for x, y in loader:
    loss = criterion(model(x), y)
    loss.backward()            # grad += 이전 grad ❗
    optimizer.step()           # 잘못된 업데이트

# 3) ✅ 의도적 사용 (gradient accumulation: 큰 batch 효과)
accum_steps = 4
optimizer.zero_grad()
for i, (x, y) in enumerate(loader):
    loss = criterion(model(x), y)
    loss.backward()            # grad 누적
    if (i + 1) % accum_steps == 0:
        optimizer.step()       # 여러 batch 합쳐서 업데이트
        optimizer.zero_grad()  # 그때 초기화

# 🔥 핵심
# zero_grad = "이전 gradient 리셋"
# 안 쓰는 경우 = "여러 batch를 일부러 합쳐서 업데이트할 때"
