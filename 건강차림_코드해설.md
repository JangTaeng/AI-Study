# 건강차림 — 코드 전체 해설

> 이 문서는 `index.html` 한 파일을 위에서 아래로 따라가며, 각 블록이 **왜 있고 무슨 일을 하는지**를 자연어로 풀어 쓴 발표용 해설서입니다.

---

## 0. 큰 그림 한 번에 보기

이 한 파일은 세 덩어리로 나뉩니다.

1. **`<head>` (메타·폰트·파비콘)** — 브라우저에게 "이 문서는 한국어이고, 이런 폰트와 색을 쓸 거야"라고 알려주는 머리말.
2. **`<style>` (CSS, 약 800줄)** — 디자인 시스템. 색·폰트·레이아웃·각 컴포넌트의 외형을 모두 여기서 정의.
3. **`<body>` + `<script>` (HTML 구조 + JS 로직)** — 실제 화면 구조와, 그 화면을 움직이는 모든 동작.

전체 앱은 **사용자 → 플랜(Project) → 대화(Conversation) → 메시지(Message)** 라는 4단 계층으로 데이터를 다루며, `localStorage`에 통째로 저장됩니다. 백엔드 없이 프런트만으로 돌아가는 **데모용 목업(mock)** 이라는 점이 핵심입니다.

## 1. `<head>` — 문서 머리말

```html
<meta charset="utf-8" />
<meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover" />
<title>건강차림 — 대화</title>
<meta name="theme-color" content="#1F6E55" />
<link rel="icon" type="image/svg+xml" href="logo_1.svg" />
```

- `charset="utf-8"` — 한글이 깨지지 않게 인코딩을 UTF-8로 지정합니다.
- `viewport` — 모바일에서 픽셀이 아니라 기기 폭에 맞춰 보이도록 하는 표준 설정. `viewport-fit=cover`는 아이폰의 노치 영역까지 화면을 채우게 합니다.
- `theme-color` — 모바일 브라우저 주소창 색을 브랜드 컬러(짙은 세이지 그린)로 물들입니다.
- `link rel="icon"` — 탭에 표시되는 파비콘을 SVG 로고로 지정.

```html
<link rel="preconnect" href="https://fonts.googleapis.com" />
<link href="...Nanum+Gothic..." rel="stylesheet" />
```

- `preconnect` — 구글 폰트 서버에 미리 연결해서 폰트 로딩 지연을 줄이는 최적화.
- 본문 글꼴로 **나눔고딕**(400/700/800)을 가져옵니다.

## 2. `<style>` — 디자인 시스템

### 2-1. CSS 변수 (`:root`)

```css
:root {
  --sage: #2D9D78;
  --sage-dark: #1F6E55;
  --coral: #FF8A65;
  --cream: #FAFAF7;
  --ink: #1A2E2A;
  ...
  --side-w: 280px;
  --proc-w: 320px;
  --top-h: 56px;
}
```

여기서 정의한 변수가 이후 CSS 전체에서 `var(--sage)` 형태로 재사용됩니다. 한 곳에서 색·폰트·치수를 관리하니까 **나중에 브랜드 색을 바꾸고 싶으면 여기 한 줄만 고치면 끝**입니다. 이게 디자인 토큰화의 핵심 이점이에요.

- `--sage` 계열: 브랜드 메인 그린.
- `--coral`: 사용자 메시지 버블 색 (포인트 컬러).
- `--cream`: 본문 배경. 흰색보다 살짝 따뜻한 톤이라 눈이 편함.
- `--ink` 계열: 글자 색을 진하기 단계별로 정의 (본문/보조/희미).
- `--line` 계열: 구분선 색.
- `--amber`, `--danger`: 경고와 위험 상태에 쓰는 강조 색.
- `--side-w`, `--proc-w`, `--top-h`: 사이드바·프로세스 패널·탑바의 고정 폭/높이.

### 2-2. 기본 리셋과 본문 스타일

```css
*, *::before, *::after { box-sizing: border-box; }
html, body { height: 100%; overflow: hidden; }
body { word-break: keep-all; ... }
```

- `box-sizing: border-box` — 모든 요소의 크기 계산을 "테두리 포함"으로 통일. 레이아웃이 훨씬 예측 가능해집니다.
- `overflow: hidden` — 페이지 전체 스크롤은 막고, 각 패널 안쪽에서만 스크롤이 일어나게 합니다(데스크톱 앱 같은 느낌).
- `word-break: keep-all` — 한국어가 단어 중간에서 어색하게 끊기지 않도록 합니다. 영어와 다르게 한국어는 **단어 단위**로 줄바꿈해야 자연스럽거든요.

### 2-3. 스크롤바 커스터마이징

```css
::-webkit-scrollbar { width: 8px; ... }
::-webkit-scrollbar-thumb { background: var(--line); ... }
```

기본 스크롤바가 너무 두껍고 투박하니까, 8px 가는 스크롤바로 바꿔서 미니멀한 분위기를 만듭니다.

### 2-4. 앱 셸 (전체 레이아웃)

```css
.app {
  display: grid;
  grid-template-columns: var(--side-w) 1fr var(--proc-w);
  grid-template-rows: var(--top-h) 1fr;
  grid-template-areas: "topbar topbar topbar" "side main proc";
  height: 100dvh;
}
```

이 부분이 **레이아웃의 뼈대**입니다.
- `grid-template-areas`로 화면을 "탑바 / 사이드·메인·프로세스" 식의 명확한 영역으로 쪼갭니다.
- `100dvh`(dynamic viewport height) — 모바일에서 주소창이 사라질 때 높이가 흔들리는 문제를 막아주는 최신 단위입니다. `100vh`보다 훨씬 안정적이에요.

### 2-5. 탑바 (`.topbar`)

화면 맨 위 56px 띠. 좌측에 로고와 현재 플랜 컨텍스트, 우측에 설정 톱니바퀴 + 사용자 아바타 드롭다운이 배치됩니다.

- `.brand-context` — 사이드바 헤더가 아니라 탑바에서도 *현재 어떤 플랜에 있는지*를 보여주기 위한 영역. 사용자 입장에선 "지금 내가 어디에 있는지"를 항상 알 수 있어야 길을 잃지 않거든요.
- `.icon-btn` — 36x36 정사각형 버튼. hover 시 살짝 배경색이 바뀌는 마이크로 인터랙션.
- `.topbar-user` — 둥근 알약 모양 버튼. 클릭하면 우측 상단 드롭다운(`.topbar-user-menu`)이 펼쳐집니다.

### 2-6. 우측 상단 드롭다운 (`.topbar-user-menu`)

```css
.topbar-user-menu {
  opacity: 0;
  transform: translateY(-6px);
  pointer-events: none;
  transition: opacity .18s, transform .18s;
}
.topbar-user-menu.is-open { opacity: 1; transform: translateY(0); pointer-events: auto; }
```

이게 드롭다운 애니메이션의 표준 패턴입니다.
- 평소엔 **투명 + 살짝 위로 떠 있고 + 클릭 차단**.
- `is-open` 클래스가 붙으면 → 불투명해지고 + 제자리로 내려오고 + 클릭 가능.

자바스크립트에서 클래스만 토글해도 부드러운 페이드가 일어나는 구조입니다. `pointer-events: none`으로 숨길 때 "안 보이는데 클릭은 막아주는" 처리가 들어가 있어야 사용자가 빈 영역을 의미 없이 누르는 일을 막을 수 있어요.

### 2-7. 사이드바 (`.side`)

좌측 280px 패널. 위에서부터 [플랜 스위처] → [새 대화 버튼] → [대화 목록] → [사용자 스위처] 순서로 쌓입니다.

- `.proj-switcher` — 현재 플랜 이름을 보여주는 큰 버튼. 누르면 아래로 플랜 목록 메뉴가 펼쳐짐.
- `.proj-menu` — 절대 위치(`position: absolute`)로 사이드바 헤더 영역에 떠 있는 드롭다운. `box-shadow`로 떠 있는 듯한 느낌을 줍니다.
- `.proj-item.active::before` — 활성 항목 왼쪽에 3px짜리 세로 강조 바가 그려지는 트릭.
- `.proj-actions` — 항목 위에 마우스를 올렸을 때만(`opacity` 애니메이션) 나타나는 ✏️/🗑️ 아이콘 그룹.

### 2-8. 대화 목록 (`.conv-list`)

- `.conv-item` 우측에 패딩을 64px 비워둔 이유 → 이 자리에 `.conv-actions`(편집/삭제 버튼)이 절대 위치로 들어오기 때문입니다. 텍스트가 버튼과 겹치지 않게 미리 자리를 예약해 둔 거예요.
- `.conv-meta` — "3분 전 · 12 메시지" 같은 부가정보를 한 줄로.
- `.inline-edit` — 제목을 더블클릭/펜클릭하면 그 자리에서 input으로 변신하는 인라인 편집 입력칸. 별도 모달 없이 자리에서 수정할 수 있어 사용감이 깔끔합니다.

### 2-9. 사용자 스위처 (`.user-switcher`, `.user-menu`)

사이드바 맨 아래에 있는 현재 사용자 카드. 클릭하면 위로 솟아오르는 메뉴(`bottom: calc(100% + 4px)`)가 떠서 다른 사용자로 전환할 수 있게 합니다. 데모니까 4명을 미리 시드 데이터로 넣어둔 거예요.

### 2-10. 채팅 메인 (`.main`)

```css
.main { display: flex; flex-direction: column; }
.chat-scroll { flex: 1; overflow-y: auto; ... }
.input-area { flex-shrink: 0; ... }
```

세로로 [공지바] → [메시지 스크롤 영역] → [입력 바] 순. `flex: 1`이 메시지 영역에만 붙어 있어서, 입력바와 공지바는 자기 크기를 유지하고 **남는 공간을 메시지 영역이 모두 차지**합니다.

#### 메시지 버블

```css
.msg.user .bubble { background: var(--coral); border-bottom-right-radius: 4px; }
.msg.bot .bubble  { background: var(--paper); border-bottom-left-radius: 4px; }
```

말풍선의 4개 모서리 중 **꼬리에 해당하는 한 모서리만** 덜 둥글게 처리해서 "누가 말한 건지" 시각적으로 자연스럽게 구분합니다. 카카오톡, 메신저 UI의 클래식 패턴이에요.

#### 점수 카드 (`.score-card`)

음식 점수를 시각화하는 카드. 좌측에 큰 숫자, 우측에 음식 이름·세부설명·점수 막대. `border-left`의 색을 `good/warn/danger` 클래스에 따라 바꿔서 "한눈에 안전/주의/위험"을 알 수 있게 했습니다.

#### 출처 칩 (`.source-chip`)과 빠른 답장 (`.quick-reply`)

응답 아래에 함께 따라오는 부속 UI들.
- 출처 칩 = "이 답변이 어떤 의료 문서에서 나왔는지" 표시. RAG의 신뢰성을 시각적으로 강조하는 핵심 요소.
- 빠른 답장 = 사용자가 다음으로 물어볼 만한 후속 질문 버튼.

#### 타이핑 인디케이터 (`.typing`)

```css
.typing span { animation: typing 1.4s infinite ease-in-out; }
.typing span:nth-child(2) { animation-delay: 0.18s; }
.typing span:nth-child(3) { animation-delay: 0.36s; }
```

세 점이 시간차로 위아래로 튀는 애니메이션. 지연(delay)을 다르게 주는 게 핵심입니다. 모두 똑같이 움직이면 깜빡거리는 것처럼 보이지만, 시차를 두면 "물결치듯" 자연스러워져요.

### 2-11. 입력 바 (`.input-wrap`)

- `:focus-within` — 안에 있는 textarea가 포커스를 받으면 입력 박스 전체에 초록 테두리와 글로우가 생깁니다. 부모 요소가 자식의 상태에 반응하는 가상 클래스.
- `textarea`의 `max-height: 160px` — 너무 길게 쓰면 일정 높이까지만 늘어나고, 그 이상부턴 내부 스크롤로 처리됩니다.
- `.send-btn:disabled` — 입력칸이 비어 있으면 전송 버튼이 회색으로 비활성화됩니다.

### 2-12. 스크롤 다운 버튼 (`.scroll-bottom-btn`)

채팅이 길어져서 위로 올라가서 읽다가, 갑자기 새 메시지가 와도 사용자는 모를 수 있죠. 그래서 **스크롤이 바닥에서 80px 이상 떨어지면** 우측 하단에 동그란 ⬇️ 버튼이 페이드인하면서 떠오릅니다. 클릭하면 부드럽게 맨 아래로 이동.

### 2-13. 처리 과정 패널 (`.proc`)

우측 320px 패널. RAG 시스템이 실제로 무슨 일을 하고 있는지를 시각화하는 발표의 **하이라이트** 영역입니다.

- `.proc-status.live` — 실시간 동작 중이면 초록 점이 박동하는 애니메이션(`livePulse`).
- `.proc-row.running` — 현재 실행 중인 작업은 배경이 옅은 초록으로 강조되고, 좌측 아이콘이 빙빙 돌아갑니다(`animation: spin`).
- `.proc-icon.done::after` — CSS만으로 그린 체크 아이콘. 5x8px 박스의 우/하단 테두리만 그리고 45도 회전시키면 ✓ 모양이 됩니다.
- `.doc-row` — 참조한 의료 문서 카드. 좌측에 3px 초록 강조 바, 하단에 관련도 게이지 막대.

### 2-14. 모달 (`.modal-backdrop`, `.modal`)

- 배경을 검게 어둡게 하면서 살짝 블러(`backdrop-filter: blur(4px)`)를 거는 게 요즘 모달의 표준입니다.
- 모달 자체는 평소엔 살짝 아래에 + 작아진 상태(`translateY(20px) scale(0.98)`)로 숨겨져 있다가, 열릴 때 제자리로 부드럽게 솟아오릅니다.
- 모달 내부는 [헤더(고정)] - [바디(스크롤 가능)] - [푸터(고정)] 3단 구조.

#### 라디오 버튼 커스터마이징

```css
.radio-pill input { position: absolute; opacity: 0; pointer-events: none; }
.radio-pill input:checked + span { ... }
```

기본 라디오 버튼은 못생기고 브라우저마다 다르게 보입니다. 그래서 진짜 input은 보이지 않게 숨기고(`opacity: 0`), 바로 옆 `<span>`에 스타일을 줘서 그게 라디오처럼 보이게 합니다. 체크 상태(`input:checked`)에 따라 옆 span의 색을 바꾸는 식. **카드 라디오**(`.card-radio`)도 같은 원리예요.

### 2-15. 토스트 (`.toast`)

화면 하단 가운데에 잠깐 떠올랐다가 사라지는 알림. `transform: translateX(-50%)`로 정확히 가운데 정렬한 뒤, `translateY` 애니메이션으로 위로 올라오는 효과를 줍니다.

### 2-16. 반응형

```css
@media (max-width: 1100px) { :root { --side-w: 240px; --proc-w: 280px; } }
@media (max-width: 900px) { .app { grid-template-areas: "topbar" "main"; } .side, .proc { display: none; } }
@media (prefers-reduced-motion: reduce) { ... }
```

- 1100px 이하: 패널들 폭을 줄여서 채팅 공간 확보.
- 900px 이하(태블릿/모바일): 양쪽 패널을 아예 숨기고 채팅만.
- `prefers-reduced-motion` — OS에서 "애니메이션 줄이기"를 켠 사용자에겐 모든 트랜지션을 거의 0초로 만들어줍니다. **접근성 표준**이에요.

## 3. `<body>` — HTML 구조

### 3-1. 탑바

```html
<header class="topbar">
  <div class="brand">
    <img class="brand-mark" src="logo_1.svg" />
    <span class="brand-name">건강차림</span>
    <div class="brand-context" id="brandContext"></div>
  </div>
  ...
</header>
```

- `id="brandContext"`는 비어 있는데, 자바스크립트의 `renderBrandContext()` 함수가 여기에 "현재 플랜 이름 + 관리 방향 태그"를 동적으로 채워 넣습니다.
- 우측엔 ⚙️ 설정 버튼과 사용자 아바타 드롭다운이 들어갑니다.

### 3-2. 사이드바

```html
<aside class="side">
  <div class="side-head">
    <button class="proj-switcher" id="projSwitcher">...</button>
    <button class="new-chat-btn" id="newChatBtn">새로운 대화 추가</button>
    <div class="proj-menu" id="projMenu" role="menu">...</div>
  </div>
  <div class="conv-list" id="convList"></div>
  <div class="side-foot">
```

### 4-4. 상태 저장/불러오기와 마이그레이션

```js
function loadState() {
  try {
    const raw = localStorage.getItem(STORE_KEY);
    if (raw) {
      const parsed = JSON.parse(raw);
      if (parsed.v === 6) { state = parsed; return; }
    }
    // v5 → v6 마이그레이션
    const v5raw = localStorage.getItem('gc_state_v5');
    if (v5raw) { ... }
    // v4 → v6 마이그레이션
    ...
  } catch(_) {}
  state = makeInitialState();
  saveState();
}
```

**마이그레이션 로직**이 이 코드의 숨은 자랑거리입니다.
- v4 시절엔 사용자가 직접 건강 정보(profile/condition/goal)를 가지고 있었어요.
- v6에서는 그 정보가 **플랜으로 이동**했습니다.
- 옛날 데이터(v4, v5)를 가진 사용자가 새 버전을 켜도, 자동으로 새 구조로 변환되도록 단계별 마이그레이션이 들어 있습니다. 데이터를 잃지 않으면서 구조를 바꾸는 표준 패턴이에요.

```js
function saveState() {
  try { localStorage.setItem(STORE_KEY, JSON.stringify(state)); } catch(_) {}
}
```

상태가 바뀔 때마다 호출되어 통째로 JSON으로 직렬화해 저장. `try/catch`로 감싼 이유는 시크릿 모드처럼 localStorage가 막혀 있는 환경을 대비해서입니다.

**이 코드의 머리가 가장 좋은 부분.** 같은 음식이라도 사용자의 관리 방향(당뇨/고혈압/체중/복합)에 따라 점수·이유·출처가 모두 달라집니다.

예를 들어 김밥은:

- 고혈압인에겐 65점(주의) — 이유: 나트륨 1,100mg
- 다이어트엔 70점(주의) — 이유: 480kcal
- 복합 만성질환엔 55점(위험) — 두 가지 모두 부담

같은 메뉴라도 **누가 먹느냐에 따라 답이 달라진다**는 개인화의 핵심을 데이터 구조로 표현한 부분입니다.


사용자가 입력한 텍스트에서 음식 키워드를 찾고, **현재 활성 플랜의 관리 방향에 맞는 응답**을 골라서 반환합니다. 이게 발표에서 "플랜 별 개인화"를 시연할 때 핵심 포인트.


당뇨 플랜에선 김밥/바나나가, 고혈압 플랜에선 라면/저염식이 첫 추천 질문으로 뜹니다. 컨텍스트 적응적 UX의 작은 디테일.

#### `renderMessage(m)` — 한 개의 메시지를 DOM 요소로

봇 메시지엔 본문 외에도:
- 음식 점수 카드
- 출처 칩
- 빠른 답장 버튼

이 동적으로 따라붙습니다. 이 셋은 **있으면 그릴 뿐, 없으면 생략**되도록 분기 처리되어 있어요.

#### `escapeHtml(s)`

사용자 입력에 `<script>` 같은 태그가 들어와도 그대로 코드로 실행되지 않게 막아주는 안전장치. **XSS 방어**의 기본.

### 4-12. 인라인 편집 (`editConvTitle`)


`done` 플래그가 중요해요. blur와 Enter가 둘 다 발생하는 케이스가 있어서, 한 번만 처리되도록 잠그는 안전장치입니다.

- `e.preventDefault()` — form의 기본 제출(페이지 새로고침)을 막음.
- 유효성 검사 — 이름 비어있으면 저장 안 함, 나이는 10~120 사이만 받음.
- 성공 시 → 저장 → 화면 갱신 → 모달 닫기 → 토스트 알림. **사용자에게 명확한 피드백**을 주는 흐름.

### 4-15. 플랜 모달 (`openProjectModal`, `saveProjectModal`)

```js
let projectModalState = { mode: 'create', pid: null };

function openProjectModal(mode, pid = null) {
  projectModalState = { mode, pid };
  if (mode === 'create') { ... }
  else { /* 'edit': 기존 값으로 폼 채우기 */ ... }
}
```

같은 모달이 **생성 모드**와 **수정 모드** 양쪽으로 동작합니다. `projectModalState`에 현재 어떤 모드인지를 저장해 두고, 저장 버튼이 눌렸을 때 분기 처리합니다.

```js
function saveProjectModal(e) {
  e.preventDefault();
  const profileVal = document.querySelector('input[name="projectProfile"]:checked')?.value || DEFAULT_PROFILE;
  if (projectModalState.mode === 'create') createProject(name, profileVal);
  else if (projectModalState.mode === 'edit') updateProject(projectModalState.pid, name, profileVal);
  closeProjectModal();
}
```

`?.` (옵셔널 체이닝) — 라디오 버튼이 하나도 선택되지 않은 경우라도 에러 없이 기본값으로 떨어지게 하는 방어 코드.

### 4-16. 입력 처리

```js
inputBox.addEventListener('input', () => { autoresize(); sendBtn.disabled = inputBox.value.trim().length === 0; });
inputBox.addEventListener('keydown', (e) => {
  if (e.key === 'Enter' && !e.shiftKey && !e.isComposing) {
    e.preventDefault();
    if (!sendBtn.disabled) sendMessage(inputBox.value);
  }
});
```

- **Enter** → 전송, **Shift+Enter** → 줄바꿈 (메신저 표준).
- `e.isComposing` 체크가 핵심입니다. **한글은 자모 조합 중에 Enter를 한 번 더 받아야 입력이 완료**되는 IME(Input Method Editor) 특성이 있어요. 이 체크가 없으면 한글 입력 도중에 메시지가 보내지는 버그가 생깁니다. 한국어 서비스를 만들 때 매우 자주 빠뜨리는 처리 — 이걸 챙겼다는 건 **한국어 UX에 대한 이해도가 있다**는 뜻이에요. 발표 때 어필 포인트입니다.

```js
function autoresize() {
  inputBox.style.height = 'auto';
  inputBox.style.height = Math.min(inputBox.scrollHeight, 160) + 'px';
}
```

textarea 높이를 내용에 맞춰 자동 조절. 한 줄이면 한 줄 높이, 여러 줄이면 늘어남. 단 160px이 한계.

### 4-17. 스크롤 다운 버튼

```js
function checkScrollPosition() {
  const distance = chatScroll.scrollHeight - chatScroll.scrollTop - chatScroll.clientHeight;
  scrollBottomBtn.classList.toggle('visible', distance > 80);
}
chatScroll.addEventListener('scroll', checkScrollPosition, { passive: true });
```

- "전체 콘텐츠 높이 − 위에서 스크롤된 만큼 − 보이는 영역 높이 = 바닥까지 남은 거리".
- 80px보다 멀면 ⬇️ 버튼이 페이드인.
- `{ passive: true }` — 스크롤 성능 최적화 옵션. 브라우저에게 "이 핸들러는 스크롤을 막지 않을 거야"라고 알려줘서 부드러운 스크롤이 보장됩니다.

### 4-18. 토스트

```js
let toastTimer = null;
function toast(text, iconText = '✓', iconBg = 'var(--sage)') {
  ...
  el.classList.add('show');
  clearTimeout(toastTimer);
  toastTimer = setTimeout(() => el.classList.remove('show'), 2400);
}
```

`clearTimeout`이 들어 있는 이유: 토스트가 떠 있는 도중에 다른 토스트가 또 뜨면, 이전 타이머를 취소하고 새로 2.4초를 카운트합니다. 짧은 시간에 여러 알림이 일어나도 마지막 메시지가 충분히 보이도록.
